Hoy, 14 de mayo, continué con el desarrollo y refinamiento de varios sistemas del proyecto. Las principales tareas completadas incluyen: la migración del parallax a SpriteRenderer para una mejor integración con el sistema de capas, la corrección del fire rate del enemigo torreta, la incorporación de una nueva mecánica de parry sobre pinchos (que además solucionó un problema con el enemigo Rhino) y el inicio del desarrollo de un `ResetManager` para reiniciar elementos del nivel al morir el jugador. A continuación, se detalla cada uno de estos avances.

## 1. Parallax con SpriteRenderer

Se retocó el sistema de fondo de los niveles. Ahora el parallax funciona con `SpriteRenderer` en lugar de materiales aplicados a objetos 3D. Esto permite que se integre correctamente con el flujo de trabajo de capas (sorting layers) implementado ayer para enemigos y objetos de gameplay. Aún quedan detalles por pulir, pero la base es sólida y permite continuar.

## 2. Corrección del fire rate de la torreta

Se detectó que el enemigo torreta no estaba utilizando correctamente la variable `fireRate` debido a un error en el código. Tras corregir la implementación y añadir un `Tooltip` como documentación auxiliar, el sistema de disparo funciona como se esperaba.

https://i.imgur.com/wI6TKav.png

## 3. Rhino fix y parry a pinchos (spikes)

Mientras se intentaba corregir al enemigo Rhino para que rebotase y cambiase de sentido al tocar pinchos, se modificó la layer de los pinchos. Esto no solo solucionó el problema, sino que además introdujo una nueva mecánica: **el parry de pinchos** (o parry parkour). El jugador ahora puede atacar los pinchos antes de caer sobre ellos, realizando un parry que le permite rebotar, similar a mecánicas presentes en juegos como Hollow Knight.

https://i.imgur.com/T7rrCYc.mp4

## 4. ResetManager para elementos de gameplay

Se comenzó a desarrollar un nuevo gestor: `ResetManager`. Su objetivo es permitir la comunicación entre el jugador y los distintos elementos del nivel (enemigos, plataformas, etc.) para que, cuando el jugador pierda todas sus vidas y reaparezca (en el inicio o en un checkpoint), el estado de esos elementos vuelva a su configuración inicial. Esto evita softlocks (por ejemplo, una plataforma one-way que quede en una posición incorrecta) o daños instantáneos al reaparecer por un proyectil disparado previamente.

**Flujo actual**:

```text
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  PlayerHealth   │     │  ResetManager   │     │  IReseteable    │
│                 │     │                 │     │  (interfaz)     │
└────────┬────────┘     └────────┬────────┘     └────────┬────────┘
         │                       │                       │
         │ onPlayerDeath (Unity Event)                   │
         │──────────────────────►│                       │
                                 │                       │
                                 │ Notifica a cada       │
                                 │ elemento registrado   │
                                 │──────────────────────►│
                                                        
                                                        ★ Cada elemento
                                                           implementa
                                                            su propio reset
                                                        
```

`PlayerHealth` emite un `UnityEvent` llamado `onPlayerDeath` justo antes de que el jugador reaparezca. `ResetManager` está suscrito a dicho evento. Al recibirlo, recorre todos los elementos registrados que implementan la interfaz `IReseteable` y llama a su método `ResetState()`. Cada elemento (enemigo, plataforma, etc.) define su propia lógica de reinicio, lo que mantiene el sistema desacoplado y extensible.

Actualmente esta implementado para los enemigos, y dos tipos de plataforma: `Pulley Platform` y `On Collision Platform`.
